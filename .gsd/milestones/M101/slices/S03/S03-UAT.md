# S03: PRD Import Bridge — UAT

**Milestone:** M101
**Written:** 2026-04-11

## UAT Type

- UAT mode: artifact-driven
- Why this mode is sufficient: The slice produces a CLI command and file output (REQUIREMENTS.md). Verification is deterministic — parse a known PRD, check DB state and rendered output. No runtime server or live UI needed.

## Preconditions

- Node.js installed, `npm install` has been run
- Real PRD fixture exists at `_bmad-output/planning-artifacts/prd.md`

## Smoke Test

Run `npx vitest run tests/import/` — all 21 tests should pass with zero failures.

## Test Cases

### 1. Parser extracts all requirements from real PRD

1. Import `parsePrd` from `src/import/prd-parser.ts`
2. Read `_bmad-output/planning-artifacts/prd.md`
3. Call `parsePrd(content, 'prd.md')`
4. **Expected:** Returns array of 22 items — 16 with class "functional" (FR1–FR16) and 6 with class "non-functional" (NFR1–NFR6). All IDs are strings without parenthetical names. Descriptions have no markdown bold markers.

### 2. Import command inserts requirements into DB

1. Create an in-memory GSD engine
2. Call the import handler with the real PRD file path
3. **Expected:** DB contains 22 requirements with status "active". REQUIREMENTS.md is rendered at `.gsd/REQUIREMENTS.md` with a markdown table containing all 22 rows.

### 3. Re-import is idempotent (no duplicates)

1. Import the same PRD file twice via the import handler
2. **Expected:** DB still contains exactly 22 requirements (no duplicates). Second import reports 0 new insertions.

### 4. Error handling for missing file

1. Call the import handler with a non-existent file path
2. **Expected:** No crash. Error notification produced via `ctx.ui.notify()`.

### 5. REQUIREMENTS.md has correct format

1. After import, read `.gsd/REQUIREMENTS.md`
2. **Expected:** File starts with `<!-- Auto-generated by /gsd import. Do not edit manually. -->`. Contains a markdown table with columns: ID, Class, Description, Status, Source. All 22 requirements are present.

## Edge Cases

### PRD with no requirement sections

1. Parse a markdown string with no "Functional Requirements" or "Non-Functional Requirements" headers
2. **Expected:** Returns empty array. Import handler reports "no requirements found".

### Malformed requirement lines

1. Parse markdown containing lines like `* Some random bullet` inside a requirements section
2. **Expected:** Malformed lines are skipped gracefully. Only properly formatted `**FRn:**` and `**NFRn:**` entries are extracted.

## Failure Signals

- Test suite failures in `tests/import/` — parser or import logic broken
- REQUIREMENTS.md not generated after import command — file write failure
- Duplicate requirements after re-import — deduplication logic broken

## Not Proven By This UAT

- Large PRD files (thousands of requirements) — performance not tested
- Non-BMAD PRD formats — parser targets specific BMAD format only
- Interactive CLI usage — tested via programmatic handler calls, not live terminal

## Notes for Tester

- The real PRD fixture at `_bmad-output/planning-artifacts/prd.md` is the authoritative test file — if it changes format, parser tests may need updating.
- REQUIREMENTS.md is auto-generated; any manual edits will be overwritten on next import.
